2.3.1. Aktivitätsdiagramme

Aktivitätsdiagramme beschreiben meist den Ablauf eines Anwendungsfalles, können allerdings auch zur Modellierung aller weiteren Aktivitäten innerhalb eines Systems verwendet werden. Dabei ist es auch möglich, asynchrone Kommunikationsmechanismen einzubinden (Signal senden/empfangen, Ausnahmebehandlung) und somit Nebenläufige Systeme darzustellen. Ein Aktivitätsdiagramm stellt eine objektorientierte Adaption eines Programmablaufplans dar.

Aufbau

Aktivität

Eine Aktivität beschreibt Abläufe, aus mehreren elementaren Aktionen. Der Ablauf kann parallelisiert und synchronisiert, sowie durch Bedingungen geteilt und wieder zusammengeführt werden. Operationen, die z. B. in Klassendiagrammen existieren können als Aktivität dargestellt werden. Der Ablauf einer Aktivität wird durch den Tokenfluss bestimmt. Ein Token ist dabei ein virtueller Koordinationsmechanismus; d.h. er beschreibt einen möglichen Ablauf einer Aktivität zur Laufzeit. Er fließt entlang der Kanten von Vorgänger- zu Nachfolgerknoten. Man unterscheidet zwischen Kontroll- und Datentoken. Kontrollknoten beschreiben eine "Ausführungserlaubnis" für den Folgeknoten und Datenknoten transportieren Daten oder Referenzen.

Beginn und Ende von Aktivitäten und Abläufen

Ein Aktivitätsablauf beginnt in einem Initialknoten, der im Diagramm als ausgefüllter Kreis dargestellt wird. Es können ein oder mehrere Initialknoten existieren, wodurch Nebenläufigkeit dargestellt werden kann. Ein Aktivitätsendknoten wird als unausgefüllter Kreis mit einem kleineren, ausgefüllten Kreis in seinem Mittelpunkt dargestellt. Er beendet alle Abläufe einer Aktivität; der erste Token, der zu ihm gelangt beendet die Aktivität. Kontrolltoken werden gelöscht, Datenknoten hingegen nicht. Pro Aktivität sind mehrere Aktivitätsendknoten erlaubt. Ein Ablaufendknoten beendet den Ablauf einer Aktivität. Er wird durch einen durchkreuzten Kreis dargestellt.

Entscheidungsknoten

Ein Entscheidungsknoten definiert alternative Zweige im Tokenfluss und stellt somit eine Art Weiche dar. Er kann auch für die Modellierung von Schleifen genutzt werden. Überwachungsbedingungen werden an den ausgehenden Zweigen notiert und wählen den Zweig aus. [else] ist dabei als Standard vordefiniert. Ein Entscheidungsverhalten kann an den Knoten annotiert werden und wird durch das Kennwort <<decisionInput>> gekennzeichnet. Es ermöglich eine genaue Spezifikation der Auswahlentscheidung.

Die Zusammenführung alternativer Abläufe erfolgt durch einen Vereinigungsknoten. Er reicht die Token so bald wie möglich an den Nachfolgerknoten weiter. Dabei kann Ein Vereinigungsknoten gleichzeitig ein Entscheidungsknoten sein.

Parallelisierungsknoten

Ein Parallelisierungsknoten modelliert eine Aufspaltung von Abläufen. Eingehende Token werden für alle Kanten dupliziert, sobald eine Überwachungsbedingung sie akzeptiert. Nichtakzeptierte Token werden aufbewahrt. Ein Synchronisierungsknoten führt nebenläufige Abläufe wieder zusammen. Die Token werden vereinigt, sobald sie an allen Kanten vorhanden sind. An einer Kante anliegende Kontrolltoken werden vereinigt, Kontrollknoten verschiedener Kanten werden vereinigt und als einziges Token weitergereicht. Datentoken werden alle weitergereicht. Parallelisierungsknoten und Synchronisierungsknoten können auch vereinigt werden.

Objektknoten

Objektknoten beinhalten Datentoken und stehen durch Objektflüsse miteinander in Beziehung. Der Inhalt ergibt sich aus einer Aktion und Eingabe für weitere Aktion. Typeingabe und Zustandsbeschränkungen sind dabei optional. Sie können als Parameter für Aktivitäten (activity parameter node) oder Aktionen (pins) dienen.
